# ADR-0012: FinOps governance for agent cost (visibility + budgeting + optimization + hybrid cross-charge + approval flow)

- **Status:** Accepted
- **Date:** 2026-05-07
- **Deciders:** DevOps + Finance + SA (tri-owned), exec sponsor consulted
- **Tags:** agents, finops, cost, governance
- **Supersedes:** —
- **Superseded by:** —

## Context

The base SDLC paid in human hours. The agent-augmented version pays in tokens. Conservative annualized cost estimates:

| Team scale                                     | Annualized agent cost        |
| ---------------------------------------------- | ---------------------------- |
| 5-person team, 30 PRs/day, mixed providers     | $20K-100K/year               |
| 15-person team, 80 PRs/day                     | $100K-500K/year              |
| Plus behavioral test suites (ADR-0011)         | +5-15% overhead              |
| Plus cost-runaway events (ADR-0009 prevention) | $0-$10K per incident, capped |

Without governance, four predictable failures:

| Failure                                                                             | Impact                                 |
| ----------------------------------------------------------------------------------- | -------------------------------------- |
| **Bill shock** — vendor invoice arrives 30 days post-spend; finance freezes program | Project cancelled mid-rollout          |
| **Premium-overuse** — Opus running tasks GLM-4-Air would handle for 1/30th cost     | 5-10× wasted spend                     |
| **Cross-charge confusion** — agent serves Feature A and Feature B; nobody pays      | Feature owners refuse to commit budget |
| **Cost-per-value invisibility** — no metric for "is this agent worth it?"           | Cannot defend program at budget review |

These hit at predictable points: bill shock at month 1-2, premium-overuse continuously, cross-charge at first cross-cutting refactor, cost-per-value at first budget review. **All four kill the program** if not designed for.

## Decision

Adopt **full FinOps governance** across three dimensions, plus hybrid cross-charge model and graduated approval flow.

### Dimension 1 — Visibility (real-time dashboard)

`docs-platform/cost-dashboard.md`, generated by linter cron job:

- Live spend: $/day per agent, per provider, vs budget burn-rate
- Per-feature cost (aggregated from feature folders' commit history)
- Per-role cost
- Per-PR cost (already in PR description, ADR-0007)
- Provider mix (anthropic vs zai split; alert on policy drift)
- Trend lines (weekly/monthly)
- Anomaly indicators (PRs/days exceeding 2σ of normal flagged)

Source: provider APIs (token usage) + commit trailer parsing + feature-folder structure.

### Dimension 2 — Budgeting (per-feature + per-team caps)

**Per-feature** in OWNERS.md:

```yaml
budget:
  monthly_token_cost: $500
  hard_cap: $750
  provider_mix_target:
    claude-opus: 30%
    claude-sonnet: 50%
    glm-4-air: 20%
```

**Per-team / org** in `docs-platform/policies/cost-governance.md`:

```yaml
budgets:
  team_orders:
    monthly: $5000
    hard_cap: $7500
  team_billing:
    monthly: $3000
    hard_cap: $4500
  platform_pool:
    monthly: $2000
    hard_cap: $3000
```

**Graduated cap enforcement:**

| Threshold                             | Action                                               |
| ------------------------------------- | ---------------------------------------------------- |
| 70% of monthly cap                    | Informational notification to role leads             |
| 85%                                   | Notify SA + DevOps                                   |
| 95%                                   | Block new non-emergency invocations                  |
| 100%                                  | All agent invocations blocked except signed override |
| Per-PR cap (e.g. $5)                  | Block PR; require approval                           |
| Per-invocation cap (e.g. 200K tokens) | Pre-flight check; reject before sending              |

Hard caps are circuit breakers; system always degrades safely (humans take over).

### Dimension 3 — Optimization (provider-mix routing)

`docs-platform/standards/agent-routing.md`:

| Artifact / task                | Preferred agent                 | Rationale                                   |
| ------------------------------ | ------------------------------- | ------------------------------------------- |
| ADR / Critical-tier            | `@claude-opus-reviewer`         | High-stakes; premium justified              |
| Internal feature initial draft | `@glm-4-air-doc-drafter`        | Cheap drafting; Opus reviews catches issues |
| Test stub generation           | `@claude-sonnet-test-automator` | Mid-tier; reliable for patterns             |
| Cross-provider advisory review | opposite provider from author   | Uncorrelated blind spots (ADR-0008)         |
| Routine link/SHA bumps         | `@glm-4-air-doc-drafter`        | Cheap; mechanical                           |
| Schema migration design        | `@claude-opus-reviewer`         | Critical correctness; premium               |

Linter validates routing as **advisory** (sometimes premium is justified for non-routine reasons), but flags routine-task-on-premium-model patterns and surfaces them in the dashboard.

**Caching and batching:**

- Cache repeated agent invocations (same prompt + context) for 24h
- Batch fan-out PRs into single agent session where possible
- Re-use prior reviews via citation rather than re-generation

### Hybrid cross-charge model

| Work category                                               | Who pays                               |
| ----------------------------------------------------------- | -------------------------------------- |
| Per-feature work (inside a feature folder)                  | Feature's team budget                  |
| Cross-cutting work (template change, ADR draft, fan-out PR) | **Platform pool** (central R&D budget) |
| Roster-management bots (reassignment, fan-out)              | Platform pool                          |
| Behavioral test suites (ADR-0011)                           | Platform pool                          |
| Cost dashboard generation, linter runs                      | Platform pool                          |
| Cross-provider advisory review                              | Reviewed feature's budget              |

**Platform pool is owned by DevOps + funded out of central engineering budget.** Acts as a buffer — features don't get penalized for benefiting from cross-cutting improvements; cross-cutting work isn't blocked by individual feature budgets. **Must be a real, funded budget line, not unbudgeted hope.**

Per-feature reports show direct spend AND "pooled benefit" (estimated value of cross-cutting work that helped the feature) for transparency.

### Graduated approval flow for overages

| Range               | Approver                                          |
| ------------------- | ------------------------------------------------- |
| Up to 110% of cap   | Role lead approves                                |
| Up to 130%          | SA + DevOps co-approve                            |
| Up to 200%          | Exec sponsor approval                             |
| > 200% or recurring | Mandatory ADR (rationale + projection adjustment) |

Each override logged with named approver and reason. Recurring overages (>2 in a quarter) trigger budget reset/adjustment via ADR.

### Cost-per-value metrics (the program's defense to leadership)

- Cost per artifact authored = $/feature / artifacts shipped
- Cost per defect prevented = $/cross-provider review × defect catch rate
- Cost per human-hour saved = $/agent task × estimated human equivalent time
- Cost per incident avoided = harder; use proxy of cost-runaway-prevention

Tracked in dashboard. Reviewed quarterly. **Without these metrics, you cannot defend the program against "just use humans."**

### Role-card budget sections

Every role-card includes:

- Monthly agent budget for the role
- Provider-mix expectation
- "When to escalate from cheap to premium" guidance
- Cost-runaway runbook entry (cross-link to ADR-0009)

### Owners

**Tri-owned policy:**

- **DevOps** — operates dashboard and enforcement
- **Finance** — sets caps and approves overages
- **SA** — sets routing policy and cost-per-value targets

## Consequences

### Positive

- Bill shock prevented — caps + dashboard catch overruns before invoice.
- Premium-overuse caught and routed to cheaper models when appropriate.
- Cross-charge model is fair (features pay direct, platform pays cross-cutting).
- Cost-per-value metrics defend the program at budget review.
- Real degradation path: hard caps cause humans to take over, not silent overspend.

### Negative / Costs

- ~3 weeks added platform work (dashboard + caps + routing + metrics + policy doc).
- Ongoing FinOps cadence ~5% of DevOps time.
- Platform pool requires explicit funding decision; political work.
- Provider mix targets create overhead per feature (ratio tracking).
- Approval flow can slow legitimate overages; tune thresholds carefully.

### Neutral

- Per-feature budgets make some teams feel constrained; the tradeoff is predictability.
- Caching can produce surprising staleness if context changes; cache invalidation needs care.

## Alternatives Considered

### A. Visibility + caps only — Rejected

Dashboard + hard caps; skip cross-charge and provider-mix policy. Cheaper to build but no defense at budget review; bill shock prevented but premium-overuse continues.

### B. Caps only — Rejected

Just hard caps; no visibility, no optimization. Survives first month; fails at scale because nobody can answer "why is spend up?"

### C. None — pay invoices, treat as overhead — Rejected

Viable only if total agent spend stays small. At 70% agent ratio across 8 roles, almost certainly not the case.

## Notes for future revision

- **Platform pool funding** is the most fragile part. If it's not a real budget line, cross-cutting work won't get done; features will refuse to absorb it; the system stalls. Get explicit exec commitment **before** Phase 0.
- **Cost-per-value metrics** evolve. Start simple; refine as data accumulates. The first quarterly review will surface measurement gaps; iterate.
- **Provider mix targets** are advisory, not enforced. If observed mix consistently violates targets, the targets are probably wrong (not the routing); recalibrate.
- **Caching staleness** can produce hard-to-debug agent failures; track cache hit rate and false-cache-hit rate; bound cache TTLs aggressively for evolving content.
- Multi-currency / multi-tenancy considerations deferred. If the org adds regional teams or external customers, revisit.
