# ADR-0002: Doc-as-source-of-truth (lifecycle in docs, ad-hoc work in issues, board derived)

- **Status:** Accepted
- **Date:** 2026-05-07
- **Deciders:** SA (lead), PO, DevOps, all roles consulted
- **Tags:** workflow, source-of-truth, foundation
- **Supersedes:** —
- **Superseded by:** —

## Context

We have lifecycle state in artifact front-matter (`status: Draft | In Review | Approved | Superseded`) plus PR review state. That covers **lifecycle state**, but not **operational state**:

- Who is _actively coding right now_ on the feature?
- What is _blocking_ the schema work?
- Which features are in _this sprint_?
- How many points did design take vs estimate?
- Where do _bugs found post-deploy_ live? (Not artifact-lifecycle events.)
- Where do _spikes_ and _investigations_ live? (Same — not artifacts.)
- Where do _ops tickets_ live? (Outage, capacity, runbook gap — not artifacts.)

The default reflex is to add GitHub Issues + a Project board on top of the doc system. **This creates a dual source of truth** for the same fact ("status of this work"), and the two surfaces drift within weeks. Concretely:

- **Mismatched status**: Doc says `Approved`, issue says **In Progress**. Bot lagged or human forgot to close. Don't know which is correct.
- **Missing artifact, present issue**: Issue closed, schema doc never updated. "Missing" by our definition. Issues celebrate completion; docs evidence it.
- **Where bugs go**: Defect post-deploy is not a phase transition. If we force it into the artifact graph, the graph becomes meaningless. If we use issues, two work surfaces exist.
- **Sprint planning**: Burndown needs estimates. Estimates in front-matter make docs a planning tool (churn risk). In issues, drift again.

We must commit to **one authoritative surface per fact**.

## Decision

**Lifecycle artifacts are the source of truth for lifecycle state. Issues are the source of truth for ad-hoc work. The board is a derived view.**

Specifically:

| Surface                            | Authoritative for                                                                        |
| ---------------------------------- | ---------------------------------------------------------------------------------------- |
| **Artifact .md + front-matter**    | `status`, `owners`, `predecessors`, `last_synced_with`, `applies_adrs`, `classification` |
| **PR on the artifact**             | Approval state, merge                                                                    |
| **GitHub Issues**                  | Bugs, spikes, ops tickets, follow-ups, infra requests                                    |
| **Project board / `dashboard.md`** | Visualization only — generated by the linter from front-matter + issues                  |

**Strict rules:**

1. **Lifecycle artifacts MUST NOT have parallel tracking issues.** Their lifecycle is the artifact + its PR. The linter rejects PRs that introduce an issue mirroring an artifact.
2. **Issues are forbidden as duplicate trackers.** "Don't open an issue to track the schema design — open a draft PR on `03-schema.md`."
3. **Estimation and sprint membership** for lifecycle artifacts go in front-matter (`effort_estimate`, `sprint`). For ad-hoc work, in issue labels.
4. **The board is read-only** to humans. The linter writes `dashboard.md` and/or pushes to GitHub Projects v2 via API.
5. **Three issue types maximum**: `bug`, `spike`, `ops`. Triage labels: `needs-triage`, `ready-for-agent`, `ready-for-human`, `wontfix`.

## Consequences

### Positive

- **No drift**: each fact has exactly one home.
- **Bugs/spikes/ops have a clean home** without polluting the artifact graph.
- **Single mental model**: "is this lifecycle work? PR. Else issue."
- **Dashboard cannot drift from sources** because it's regenerated.
- **Audit trail** is git history of artifacts — first-class, immutable.

### Negative / Costs

- **PMs accustomed to issue-driven planning** must re-learn that lifecycle work is in PRs, not issues. Cultural friction in the first month.
- **Reporting tools** (e.g. velocity dashboards in Jira/Linear) don't natively read markdown front-matter. Linter must export structured data (JSON, CSV) for downstream tools.
- **Sprint planning meetings** look at a generated dashboard, not a familiar Jira board. Some PMs will reject this; need to negotiate or build a Projects v2 view that pulls from front-matter.
- **GitHub-specific tooling lock-in** somewhat: porting to GitLab requires re-implementing the dashboard exporter and issue rules.

### Neutral

- Issues remain a useful tool — they're not deprecated, just scoped. Bug triage, oncall handoffs, spike outcomes all benefit from the issue model.
- The dashboard's depth is a tunable: start with `docs/dashboard.md` (markdown table, generated). Add Projects v2 integration only if PMs require visual UI.

## Alternatives Considered

### A. Issue-as-source-of-truth, docs are deliverables (PM-style) — Rejected

Every artifact has a parent issue; docs are issue deliverables. Closing the issue requires the doc artifact merged. **Inverts the design** — docs become downstream. This works but contradicts the project framing ("share .md files"); we'd be redoing earlier decisions. Re-evaluate only if the team's existing PM tooling makes it the lower-friction path.

### B. Both surfaces independent, manually synced — Rejected

What most teams accidentally do. Drift within 2-4 weeks. **Incompatible with "no missing"** — drop the strict claim if this path is taken.

### C. Just docs, no issues — Rejected

Bugs and spikes don't fit the artifact lifecycle. They'd leak into Slack and disappear. Some operational work genuinely doesn't belong in an SDLC graph; an issue tracker handles it well.

## Notes for future revision

- If the team grows past ~30 people and reporting needs become heavier, consider promoting the dashboard to a real BI surface (e.g. Metabase querying the linter's exported JSON). Don't try to make GitHub Projects v2 fit a 30-person org.
- Watch for "creeping issue tracking": every time a team member opens an issue that should have been a PR, that's a culture signal — refresh the role-card guidance, not the rule.
